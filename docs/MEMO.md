## fix/mcp-phase-0-1

### Resumen
Ejecucion de las Fases 0.1, 0.2, 0.3, 1.1, 1.2, 1.3, 1.4 y 2.1 del plan MCP. Primero se congelo la superficie canonica: `HTTP MCP` y `stdio MCP` ya quedan separados explicitamente en codigo y documentacion. Despues se fijo el contrato minimo obligatorio de respuesta en la capa de `query_audit`: toda entrada puede reconstruirse con `request_id`, `tool_name`, `sources`, `confidence`, `completeness` y `verified`, incluyendo el caso negativo explicito cuando no hay base suficiente o la llamada `stdio` no queda verificada. Luego se fijo una suite de regresion para preguntas de alto riesgo y se endurecio el contrato de verdad/completitud de `modelos`: las respuestas con base insuficiente ahora exponen `NO VERIFICADO`, y un modelo sin instrucciones/casillas oficiales suficientes o con metadata operativa inferida queda marcado como `parcial` y no verificado. Despues se cubrio la superficie HTTP MCP prioritaria con audit trail E2E por endpoint real, incluyendo el alias `/v1/legislacion/buscar` y la familia critica de `legislacion`, `doctrina` y `modelos`. Luego se hizo auditable la superficie `stdio`: cada `tools/call` genera un `request_id` real, lo propaga a los subrequests REST y persiste una fila stdio correlada con el audit trail y grounding reales del runtime. A continuacion se fijo con cobertura HTTP que `/v1/ai/query-audit` expone esos campos de confianza/completitud con valores reales del runtime de `/v1/consulta`, por lo que Fase 1.3 se cierra como validacion de contrato implementado, no como refactor adicional. Luego Fase 1.4 persiste la respuesta final suficiente para reconstruccion: `query_audit_log` ya guarda `response_payload` tanto para `/v1/consulta` como para `stdio`, con migracion Alembic propia y reparacion runtime de esquemas legacy en SQLite/tests. Finalmente, Fase 2.1 endurece el lifecycle DB para que el cierre de sesion sea terminal y deje de ocultar reusos invalidos: `consulta` vuelve a abrir sesion solo donde la necesita, `legislacion` y `jurisprudencia` dejan de usar `next(get_db())` fuera del ciclo DI de FastAPI, `modelos` ya no deja huecos de auditoria en respuestas vacias de campana, el contrato `query_audit_log` queda tambien respaldado por Alembic y las citas del reranker se serializan con score normalizado y `source_url` coherente. La suite real del slice queda alineada con el contrato actual del API mediante ajustes minimos en routers montados, schemas, fixtures SQLite y guards de integridad Alembic.
