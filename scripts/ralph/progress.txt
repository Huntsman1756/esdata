Story: A-01 Fix Decimal serialization bug in /v1/legislacion/buscar (P0 bloqueante)
Branch: fix/audit-critical
Verification run: curl /v1/legislacion/buscar?q=IVA (200 + resultados LIVA), curl /v1/consulta?q=IVA (200), curl /v1/legislacion/LIVA/articulos/1 (200 + boe_reference) = regression check
Result: PASS
Fix applied:
- apps/api/routers/buscar.py: replaced `return JSONResponse(content=result)` with `return result` in both buscar_legislacion and buscar_legislacion_hybrid. FastAPI's default jsonable_encoder handles Decimal; JSONResponse uses stdlib json.dumps which does not.
- apps/api/routers/consulta.py line 1148: removed redundant `logger = logging.getLogger(__name__)` reassignment. The module-level logger at line 24 was being shadowed because Python treats any name assigned anywhere in a function as local in the entire function → UnboundLocalError when line 1010 used it.
Learnings:
- **Pattern:** Never wrap SQLAlchemy result dicts in `JSONResponse(content=...)` when they contain numeric types. Use `return result` and let FastAPI's jsonable_encoder handle Decimal→float.
- **Gotcha:** Python function scope — reassigning a module-level name inside a function (even inside try/except) makes it local for the whole function. If you need a logger variant, name it differently.
- **Test:** /v1/consulta was a separate bug in the same audit finding; "Decimal serialization" was only half the story — the traceback mentioned line 161 of buscar.py; but /v1/consulta threw UnboundLocalError. Both now fixed.
